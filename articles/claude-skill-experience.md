## 我的痛点：重复，无尽的重复

### 问题是怎么开始的

我是一个全栈开发者，日常工作涉及前端、后端、DevOps 多个领域。使用 Claude Code 之前，我的典型工作流是这样的：

**场景一：代码提交**

```
我：帮我生成 commit 信息
Claude：好的，请问你们团队的 commit 规范是什么？
我：我们用 conventional commits，格式是 type(scope): description
Claude：好的，那 type 有哪些？
我：feat、fix、docs、style、refactor、test、chore
Claude：好的，scope 呢？
我：前端是 fe，后端是 be，通用是 common...
（来来回回好几轮）
```

**场景二：代码审查**

```
我：帮我 review 这段代码
Claude：好的，我会检查代码质量。请问有特别关注的点吗？
我：检查性能、安全、可读性
Claude：好的，你们团队的代码规范是什么？
我：我们用 ESLint + Prettier，还有内部的一份代码审查清单...
（每次都要重新说明）
```

**场景三：写技术文档**

```
我：帮我写一篇技术文章
Claude：好的，请问你想写什么主题？
我：关于 Redis 缓存的最佳实践
Claude：好的，文章风格呢？目标读者呢？字数要求呢？
我：面向中高级开发者，3000 字左右，实战风格...
（每次都要从零开始沟通）
```

### 问题的本质

我发现，**80% 的对话内容都是重复的**——描述需求背景、说明团队规范、定义输出格式。真正有价值的内容只有 20%。

这就像每次点外卖都要先给餐厅讲一遍「我不吃辣、少油、多加葱」——效率怎么可能高？

## 转机：发现 Claude Skill

### 初次尝试

第一次接触 Skill 是看到官方文档里的示例。抱着试试看的心态，我创建了一个最简单的 Skill：

```
.claude/skills/commit/
└── SKILL.md
```

```markdown
# commit

生成符合团队规范的 commit 信息。

## 规范要求

- 格式：type(scope): description
- type：feat | fix | docs | style | refactor | test | chore
- scope：fe | be | common | infra
- description：中文，不超过 50 字

## 执行流程

1. 分析当前改动
2. 识别改动类型和范围
3. 生成符合规范的 commit 信息
```

然后，神奇的事情发生了：

```
/commit

Claude：正在分析改动...
feat(fe): 添加用户登录页面的表单验证功能
```

**一次命令，搞定。** 不需要重复说明规范，不需要多轮对话。

### 效率提升的数据

我把这个方式推广到了更多场景，一个月后的数据：

| 场景 | 之前平均耗时 | 使用 Skill 后 | 效率提升 |

| 生成 commit | 3-5 分钟 | 30 秒 | **10 倍** |
| 代码审查 | 10-15 分钟 | 3-5 分钟 | **3 倍** |
| 写技术文档 | 2-3 小时 | 40-60 分钟 | **2.5 倍** |

整体来看，**我的开发效率至少提升了 50%**。

## 实战：我开发的 5 个核心 Skill

### Skill 一：代码审查器

**痛点**：每次代码审查都要说明关注点、团队规范

**Skill 结构**：

```
.claude/skills/code-review/
├── SKILL.md
├── templates/
│   └── report.md
└── references/
    └── checklist.md
```

**核心配置**：

```markdown
# code-review

代码审查助手 - 按照团队规范审查代码变更。

## 审查维度

1. **代码质量**：可读性、可维护性、设计模式
2. **性能问题**：时间复杂度、内存泄漏、N+1 查询
3. **安全风险**：SQL 注入、XSS、敏感信息暴露
4. **团队规范**：命名规范、注释规范、文件组织

## 输出格式

使用 templates/report.md 输出结构化报告。

## 参考规范

严格遵循 references/checklist.md 中的审查清单。
```

**效果**：代码审查时间从 15 分钟降到 5 分钟，漏检率明显下降。

### Skill 二：测试生成器

**痛点**：写测试用例枯燥，覆盖率总是上不去

**Skill 结构**：

```
.claude/skills/test-gen/
├── SKILL.md
└── templates/
    ├── unit-test.md
    └── integration-test.md
```

**核心配置**：

```markdown
# test-gen

测试用例生成器 - 自动生成全面的测试用例。

## 测试策略

1. **边界值测试**：最大值、最小值、临界值
2. **异常测试**：空值、错误类型、超限值
3. **逻辑分支**：覆盖所有 if/else 分支
4. **集成测试**：模块间交互的正确性

## 输出要求

- 使用 Jest/Vitest 语法
- 覆盖率目标：80%+
- 包含 describe/it 结构
- 每个测试有清晰的描述
```

**效果**：测试覆盖率从 60% 提升到 85%，测试编写时间减少 70%。

### Skill 三：API 文档生成器

**痛点**：API 文档总是和代码不同步

**Skill 结构**：

```
.claude/skills/api-doc/
├── SKILL.md
└── templates/
    └── openapi.md
```

**核心配置**：

```markdown
# api-doc

API 文档生成器 - 从代码自动生成 OpenAPI 文档。

## 文档规范

- 遵循 OpenAPI 3.0 规范
- 包含请求/响应示例
- 说明错误码含义
- 标注鉴权要求

## 自动触发

TRIGGER when: 代码中新增或修改了 API 路由定义
```

**效果**：文档和代码保持同步，文档维护成本降低 90%。

### Skill 四：错误诊断器

**痛点**：遇到报错不知道从哪里开始排查

**Skill 结构**：

```
.claude/skills/debug/
├── SKILL.md
└── references/
    └── common-errors.md
```

**核心配置**：

```markdown
# debug

错误诊断助手 - 分析错误信息并提供解决方案。

## 诊断流程

1. 解析错误堆栈
2. 定位问题代码
3. 分析可能原因
4. 提供修复建议
5. 给出预防措施

## 常见错误库

参考 references/common-errors.md 中的常见错误和解决方案。
```

**效果**：问题定位时间从平均 30 分钟降到 10 分钟。

### Skill 五：技术写作助手

**痛点**：写技术博客每次都要想结构、查格式

**Skill 结构**：

```
.claude/skills/tech-essay/
├── SKILL.md
├── templates/
│   ├── tutorial.md
│   ├── case-study.md
│   └── experience.md
└── references/
    ├── title-guide.md
    └── review-guide.md
```

**核心配置**：

```markdown
# tech-essay

技术文章写作助手 - 提供结构化的写作指导。

## 文章类型

- 技术教程：帮助读者学会某项技术
- 实战案例：展示具体问题的解决方案
- 项目经验：分享项目中的经验教训
- 技术分析：深入分析技术方案

## 写作流程

1. 明确目标读者和核心观点
2. 选择合适的文章结构模板
3. 填充内容并打磨细节
4. 质量检查和评审

## 输出规范

遵循五大爆款标题公式，确保文章质量。
```

**效果**：写一篇技术文章的时间从 3 小时降到 1 小时，文章质量也更稳定。

## 踩过的坑

### 坑一：TRIGGER 条件写得太宽泛

**问题描述**：

我一开始给 `code-review` 写的触发条件是：

```markdown
TRIGGER when: code changes detected
```

结果每次保存文件都会触发审查，烦得要死。

**解决方案**：

精确触发条件：

```markdown
TRIGGER when: user explicitly asks for code review or says "审查代码"
DO NOT TRIGGER when: normal coding workflow
```

**启示**：**TRIGGER 条件宁可窄，不要宽**。频繁的自动触发会打断工作流。

### 坑二：模板加载了太多内容

**问题描述**：

我在 `references/` 目录放了一个 5000 字的代码规范文档，每次执行 Skill 都要加载，响应变慢很多。

**解决方案**：

把大文档拆分成多个小文件，按需引用：

```
references/
├── naming.md      # 命名规范（500 字）
├── comments.md    # 注释规范（300 字）
└── structure.md   # 结构规范（400 字）
```

在 SKILL.md 中说明「根据审查类型加载对应规范」。

**启示**：**参考文档要精简，大文档会影响响应速度**。

### 坑三：没有版本控制

**问题描述**：

团队里每个人都自己维护 Skill，结果版本不一致，A 的 Skill 在 B 那里跑不了。

**解决方案**：

把 `.claude/skills/` 纳入 Git 管理：

```bash
git add .claude/skills/
git commit -m "feat: add team shared skills"
```

**启示**：**Skill 是团队的资产，必须纳入版本控制**。

### 坑四：过度设计

**问题描述**：

我曾经想做一个「全能 Skill」，包含代码审查、测试生成、文档生成所有功能。结果：

- 配置文件 2000 行，难以维护
- 执行时间很长，体验很差
- 出问题难以定位

**解决方案**：

遵循单一职责原则，一个 Skill 做一件事：

```
skills/
├── code-review/    # 只做代码审查
├── test-gen/       # 只生成测试
├── api-doc/        # 只生成文档
└── debug/          # 只诊断错误
```

**启示**：**Skill 的粒度要小，职责要单一**。

## 最佳实践总结

### 识别「可 Skill 化」的场景

符合以下特征的任务，都值得封装成 Skill：

| 特征 | 说明 | 例子 |
|------|------|------|
| 高频重复 | 每天都会做 | 生成 commit、代码审查 |
| 固定流程 | 步骤确定 | 写技术文档、API 文档 |
| 有规范约束 | 需要遵循特定规范 | 代码风格、commit 规范 |
| 多轮对话 | 和 AI 来回沟通多次 | 需求澄清、格式纠正 |

### Skill 开发的五个步骤

1. **明确场景**：这个 Skill 解决什么问题？
2. **定义流程**：执行的步骤是什么？
3. **整理规范**：需要遵循哪些规范？
4. **创建模板**：输出格式是什么？
5. **验证迭代**：实际使用，持续优化

### Skill 的目录结构建议

```
.claude/skills/
├── skill-name/
│   ├── SKILL.md           # 主配置（必需）
│   ├── templates/         # 输出模板
│   │   └── *.md
│   └── references/        # 参考文档
│       └── *.md
```

### 团队推广建议

1. **从高频场景开始**：先做 commit、code-review 这种每天都用的
2. **提供使用文档**：每个 Skill 配一个简单的 README
3. **收集反馈迭代**：定期收集团队意见，持续优化
4. **建立分享机制**：鼓励团队成员贡献自己的 Skill

## 写在最后

### 核心要点

回顾这半年的 Skill 实践，最想分享的是：

1. **Skill 的本质是「固化最佳实践」**，把一次性的沟通成本变成一次性的配置成本
2. **效率提升来自减少重复**，识别那些每天都在重复的事情，把它们 Skill 化
3. **好的 Skill 是迭代出来的**，先用起来，在使用中持续优化

### 如果重来一次

如果重来一次，我会：

1. **更早开始**：不用等到「完美」的方案，先从最简单的开始
2. **更小粒度**：一个 Skill 做一件事，不要贪多求全
3. **更早推广**：让团队尽早参与，集思广益

### 给读者的建议

**不要等到「完美时机」**。今天就可以创建你的第一个 Skill，哪怕只有几行配置。效率提升，从一个小小的 `/command` 开始。
