# 文章标题生成规范

本文档定义技术文章标题的生成规范，包含5大爆款公式。

---

## 核心原则

**好标题 = 准确 + 吸引 + 价值**

1. **准确**：标题与内容相符，不做标题党
2. **吸引**：激发读者点击和阅读兴趣
3. **价值**：让读者预期能获得什么

---

## 五大爆款公式

### 公式一：数字量化型

**结构**：`[数字] + [单位/量词] + [核心主题] + [结果/效果]`

**原理**：数字具体、可信、易记，让读者对内容有明确预期

**示例**：

| 普通标题 | 爆款标题 |
|----------|----------|
| React 性能优化技巧 | 5 个 React 性能优化技巧，让页面快 3 倍 |
| 如何学习 TypeScript | 30 天从零掌握 TypeScript 的学习路径 |
| 微服务架构实践 | 从单体到微服务：我们拆分了 50 个服务的血泪史 |

**适用场景**：教程类、经验类、技巧类文章

---

### 公式二：痛点解决型

**结构**：`[痛点问题] + [解决方案] + [效果]`

**原理**：直接点出读者痛点，给出解决方案，激发阅读欲望

**示例**：

| 普通标题 | 爆款标题 |
|----------|----------|
| Redis 入门教程 | 还在为缓存穿透发愁？一文彻底搞懂 Redis 缓存策略 |
| MySQL 索引介绍 | 查询慢？掌握这 3 种 MySQL 索引优化技巧，性能提升 10 倍 |
| Docker 使用指南 | 本地环境又炸了？Docker 一键搞定开发环境 |

**适用场景**：实战类、问题解决类文章

---

### 公式三：对比反差型

**结构**：`[方案A] vs [方案B]：[对比维度] + [结论]`

**原理**：通过对比制造冲突，帮助读者做决策

**示例**：

| 普通标题 | 爆款标题 |
|----------|----------|
| React 和 Vue 的区别 | React vs Vue：2024 年前端框架终极选型指南 |
| MySQL 和 PostgreSQL 对比 | MySQL vs PostgreSQL：谁才是后端开发的最佳选择？ |
| 微服务与单体架构 | 微服务 vs 单体架构：90% 的项目都不该上微服务 |

**适用场景**：技术选型、调研分析类文章

---

### 公式四：揭秘解密型

**结构**：`揭秘/深扒/解析 + [技术/方案] + [核心发现]`

**原理**：制造神秘感和期待感，激发好奇心

**示例**：

| 普通标题 | 爆款标题 |
|----------|----------|
| React Hooks 原理 | 深入 React Hooks 底层：原来 useState 是这样实现的 |
| V8 垃圾回收机制 | 揭秘 V8 引擎：JavaScript 内存管理的前世今生 |
| 大厂架构设计 | 解密阿里双 11：如何扛住 50 万 QPS 的技术架构 |

**适用场景**：原理讲解、源码分析、深度研究类文章

---

### 公式五：结果导向型

**结构**：`[时间/过程] + [做到了什么] + [核心成果]`

**原理**：用结果说话，让读者相信"我也能做到"

**示例**：

| 普通标题 | 爆款标题 |
|----------|----------|
| 接口性能优化 | 我是如何把接口响应时间从 5 秒优化到 50 毫秒的 |
| 从零搭建监控系统 | 从 0 到 1：我们是如何搭建前端监控体系的 |
| 重构项目经验 | 重构 20 万行代码后，我总结了这 8 条经验 |

**适用场景**：项目经验、复盘总结、实战案例类文章

---

## 公式选择指南

| 文章类型 | 推荐公式 | 备选公式 |
|----------|----------|----------|
| 技术教程 | 数字量化型 | 痛点解决型 |
| 实战案例 | 结果导向型 | 痛点解决型 |
| 项目经验 | 结果导向型 | 数字量化型 |
| 技术分析 | 对比反差型 | 揭秘解密型 |

---

## 标题自检清单

写完标题后，对照检查：

- [ ] 是否准确反映文章内容？
- [ ] 是否包含核心关键词？
- [ ] 是否有吸引力但不做标题党？
- [ ] 长度是否控制在 30 字以内？
- [ ] 是否让读者预期到价值？

---

## 常见错误

### ❌ 错误示例

1. **过于宽泛**
   - "JavaScript 教程" → 没有明确价值点

2. **标题党**
   - "震惊！这个技术太牛了" → 内容不符预期

3. **过于技术化**
   - "基于 React 18 Concurrent 特性的状态管理方案" → 普通读者看不懂

4. **没有价值点**
   - "浅谈 Redis" → 读者不知道能获得什么

### ✅ 正确示例

1. **具体明确**
   - "Redis 实战：5 种数据结构的使用场景与性能对比"

2. **价值清晰**
   - "30 分钟掌握 Redis 缓存策略，解决 90% 的缓存问题"

3. **易懂有料**
   - "图解 React 18：并发渲染让页面快在哪？"

4. **直击痛点**
   - "线上事故复盘：Redis 缓存雪崩如何避免？"

## 标题生成流程

当需要为文章生成标题时：

1. **确定文章类型** → 选择对应公式
2. **提取核心要素** → 技术、数字、痛点、结果
3. **套用公式生成** → 按公式结构组合
4. **检查优化** → 对照自检清单调整
5. **A/B 测试** → 生成 2-3 个候选标题，选择最优

---

## 示例：完整的标题生成过程

**文章内容**：介绍如何使用 WebSocket 实现实时聊天功能

### Step 1：确定类型
→ 实战案例类 → 推荐结果导向型

### Step 2：提取要素
- 技术：WebSocket
- 场景：实时聊天
- 结果：完整实现、可运行代码

### Step 3：套用公式
```
从 0 到 1：用 WebSocket 实现一个实时聊天系统
```

### Step 4：检查优化
- 是否准确？✅
- 有关键词？✅ WebSocket、实时聊天
- 有吸引力？可以更具体

优化后：
```
从 0 到 1：用 WebSocket 10 分钟实现一个实时聊天系统
```

### Step 5：生成候选
1. 从 0 到 1：用 WebSocket 10 分钟实现实时聊天系统
2. 还在用轮询？WebSocket 让你的聊天系统实时响应
3. 实战 WebSocket：手把手教你搭建实时聊天功能

**选择**：根据发布平台和目标读者选择最合适的
